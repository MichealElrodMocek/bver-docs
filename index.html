Frame the platform. Then ship plugins.

BVER is a local-first runtime + plugin ecosystem for engineering platforms. Start with an on-prem Sandbox deployment, define your domain objects and automations, then compose capabilities via plugins (simulators, adapters, transmitters, solvers, and orchestrators).

Persistent base objects + domain kinds. Your core repo owns identity and semantics.

  • Stable vocabulary and kinds
  • Versioning + migrations
  • Compatibility contracts

Automations

Triggers, gates, and scheduled workflows that turn platform events into actions.

  • Event-driven triggers

Plugins

Plugin ecosystem (MVP)

Canonical plugin types

  • Simulator: scoped simulation context -> result
  • Adapter: external <-> internal representations
  • Transmitter: irreversible external action
  • Retriever: pulls external data into platform
  • Generator/Solver: inputs -> solution objects
  • Perception pipeline: pluggable CNN/GNN-style stages
  • Orchestrator: gated multi-step workflows
  • Validator/Policy: constraints + compliance evidence
  • Enricher: adds metadata/context without identity changes

Local plugin lifecycle

  • Install: `bver plugin add` from a path or package
  • Pin: explicit versions in a local lockfile
  • Run: capabilities exposed to the runtime by contract
  • Audit: every run emits consistent logs + artifacts

Standard BVER system layout

Core repo (your app)

  • Domain objects + kinds + migrations
  • Automations, triggers, and policies
  • Orchestrators that compose plugin capabilities
  • Local environment + deployment topology

Recommended: one primary repo that defines the vocabulary and owns identity.

Plugin repos (capabilities)

  • Group by functional directive: simulators, solvers, orchestrations
  • Group adapters by provider: e.g., "all MSAL services" together
  • Ship contracts + tests + minimal runtime glue
  • Stay replaceable: no core-domain ownership

See BVER_SYSTEM_LAYOUT.md for a concrete repo tree.
